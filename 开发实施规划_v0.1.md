# 《下笔有元》开发实施规划 v0.1

> 生成时间：2026-05-19
> 当前阶段：开发规划
> 基础需求文件：`下笔有元_产品后台与OEM系统详细规划.md`

## 1. 已确认的一期决策

1. 后端底座使用 CloudBase。
2. 一期接真实微信支付，不做纯模拟支付上线。
3. OEM 一期不直接铺开，先做一个主小程序验证完整商业闭环。
4. 一期先做总后台，由总后台控制主小程序。
5. 代理商后台、一键 OEM、多小程序代发布后置。
6. 用户端前台统一称呼“智多星”，不出现“AI 分身”等说法。
7. 代理商后台未来也不得人工开通年卡、解锁单封信件或修改订单权益。

## 2. 一期目标

一期目标不是把完整 OEM 平台一次性做完，而是先验证：

> 用户能在主小程序里通过智多星完成一次销售信生成，并通过手机号、支付、订单流水、权益流水完成可收费闭环。

成功标准：

1. 新用户可不绑定手机号进入体验。
2. 用户完成信息采集后可生成销售信。
3. 首次完整销售信免费，但查看全文前必须绑定手机号。
4. 第二次生成只能看预览，查看全文必须单封解锁或购买年卡。
5. 真实微信支付成功后，系统自动发放权益。
6. 支付回调失败或重复回调不会错误发放权益。
7. 总后台能查看用户、销售信、订单、权益、支付回调和核心配置。
8. 总后台能关闭主小程序的新生成、支付或 AI 能力。

## 3. 实时通话感方案判断

“实时通话感”不能只看一个大模型，还取决于：

- 小程序端音频采集能力；
- RTC 或 WebSocket 传输延迟；
- ASR 流式识别质量；
- VAD 断句；
- 用户打断；
- LLM 首 token 延迟；
- TTS 首包延迟；
- 播放与字幕同步；
- 弱网表现；
- 成本熔断。

### 3.1 一期推荐路线

一期正式开发先采用：

> 通话感界面 + 按住说话互动。

原因：

1. 项目一期最重要的是验证“用户愿意说完、生成后愿意绑定手机号、二次愿意付费”的商业闭环。
2. 按住说话链路更短：录音、上传、ASR、模型回复、TTS 播放，CloudBase 更容易承接。
3. 成本和失败边界更可控，每轮语音都有明确输入输出，便于保存、重试和排查。
4. 用户端仍然设计成“接通智多星”的通话感页面，只是底层交互先用按住说话。
5. 实时通话能力作为 POC 并行验证，等支付和销售信闭环稳定后再切换。

### 3.2 模型候选

第一梯队候选：

1. **腾讯 TRTC AI 实时对话**
   - 定位：实时音视频通话链路和 AI 对话编排。
   - 适合：微信小程序、CloudBase、真实通话感、快速验证。
   - 风险：要确认小程序实时音视频类目和权限；模型写作质量要实测。

2. **阿里 Qwen3.5-Omni-Plus-Realtime / Qwen-Omni-Realtime**
   - 定位：原生实时音视频聊天模型，WebSocket 接入，文本和音频输出能力完整。
   - 适合：作为“国内端到端语音模型效果”的重点 POC。
   - 风险：小程序端不宜直接连模型，需要 CloudBase 服务端中转；成本和延迟必须实测。

3. **火山引擎实时对话式 AI + 豆包/方舟模型**
   - 定位：RTC + ASR + LLM + TTS 的组合方案。
   - 适合：想接近豆包式语音体验时作为备选 POC。
   - 风险：和 CloudBase、微信支付、总后台不是同一生态，工程集成成本更高。

4. **百度智能云多模态实时互动**
   - 定位：开箱式实时互动框架，支持小程序、打断、VAD、多模型接入。
   - 适合：做横向评测。
   - 风险：对本项目的微信/CloudBase/支付主链路不是最短路径。

### 3.3 一期执行建议

不要一开始把“实时通话”写死到一期主链路。

一期做一个 `voice_gateway` 抽象：

- `startSession`
- `uploadVoiceMessage`
- `receiveTranscript`
- `receiveAssistantAudio`
- `retryLastTurn`
- `endSession`
- `fallbackToText`

一期主链路：

1. 小程序端按住说话录音；
2. 上传录音到 CloudBase；
3. 服务端调用 ASR 得到文字；
4. 智多星编排服务生成追问或确认；
5. 服务端调用 TTS 得到音频；
6. 小程序播放智多星回复，并展示文字字幕。

并行做两个 POC：

1. **POC A：腾讯 TRTC AI 实时对话**
   - 验证小程序端接入、打断、字幕、延迟、CloudBase 服务端控制。

2. **POC B：Qwen-Omni-Realtime**
   - 验证中文销售访谈的理解质量、追问能力、首包延迟、TTS 自然度。

验收指标：

- 按住说话松手后到智多星开始回应：目标小于 3 秒，可接受上限 5 秒。
- 支持本轮识别错误后重新录音。
- 支持实时字幕。
- 支持失败时降级到“按住说话 + 文字回复/语音播报”。
- 单次 5 分钟访谈成本可被后台统计。

## 5. 技术架构建议

### 5.1 前端

- 微信小程序：原生小程序或 TDesign Miniprogram。
- 总后台 Web：React + Ant Design 或 Vue + Element Plus，优先选择团队更熟的栈。
- 后台部署：CloudBase 静态托管或 CloudBase Hosting。

### 5.2 后端

推荐 CloudBase 组合：

- 云函数：轻量业务 API、微信登录、手机号、支付回调、配置读取。
- 云托管：AI 编排、长耗时生成、实时语音网关、需要长连接的服务。
- 云数据库：核心业务数据。
- 云存储：录音、导出 PDF、图片素材。
- 定时触发器：订单补偿、超时任务、成本统计。

注意：

- 实时语音和长连接不要强塞普通云函数，优先用云托管。
- 支付回调、权益发放要保持幂等。
- AI Key、微信支付配置必须只存在服务端。

## 6. 核心数据模型一期版

一期就要保留 `tenant_id`，即使目前只有主小程序。

核心表：

- `tenant`
- `admin_user`
- `app_config`
- `user`
- `user_tenant_identity`
- `phone_binding`
- `conversation_session`
- `transcript_segment`
- `memory_profile`
- `sales_letter`
- `sales_letter_version`
- `order`
- `payment_notify_log`
- `entitlement_ledger`
- `ai_provider_config`
- `ai_call_log`
- `audit_log`

关键约束：

- `order.order_no` 唯一。
- `order.wx_transaction_id` 唯一，可为空。
- `entitlement_ledger.source_order_id` 对支付权益必须唯一。
- `payment_notify_log` 记录所有回调原文和处理结果。
- `sales_letter` 是否可看全文由权益服务计算，不靠前端传参。

## 7. 开发阶段拆分

### 测试号开发策略

正式小程序未申请下来前，可以先用微信小程序测试号开发。

测试号阶段适合先做：

- 小程序页面和交互；
- CloudBase 基础工程；
- 用户端路由；
- 按住说话录音、上传、ASR、TTS 播放链路；
- 智多星对话状态机；
- 销售信生成流程；
- 我的销售信、我的档案基础页；
- 总后台基础功能；
- 数据库集合和权限规则；
- 订单与权益的模拟状态流转。

测试号阶段不要把以下能力当作最终验收：

- 真实微信支付；
- 正式手机号授权；
- 订阅消息触达；
- 小程序审核和发布；
- 依赖正式 AppID、商户号、类目或认证主体的能力。

代码要求：

1. `appid`、CloudBase 环境 ID、微信支付商户号、回调地址都必须走环境配置。
2. 支付服务要支持 `mock` 和 `wechat` 两种模式。
3. 手机号服务要支持 `mock` 和 `wechat` 两种模式。
4. 切正式小程序时，只替换配置，不重写业务流程。

### 阶段 0：项目初始化与技术验证

目标：把最不确定的风险提前打掉。

任务：

- 创建小程序工程；
- 创建 CloudBase 环境；
- 创建总后台工程；
- 使用测试号先跑通开发主链路；
- 跑通微信登录；
- 跑通手机号授权；
- 跑通真实微信支付测试链路；
- 跑通按住说话录音上传、ASR、智多星回复、TTS 播放；
- POC 腾讯 TRTC AI 实时对话；
- POC Qwen-Omni-Realtime；
- 确定一期 ASR、TTS、主模型供应商和降级方案。

验收：

- 真机可登录；
- 真机可支付 0.01/测试商品；
- 支付回调能写入数据库；
- 按住说话方案可在真机跑通；
- 生成一份 POC 延迟/成本/体验记录。

### 阶段 1：用户闭环

目标：主小程序跑通首次免费和二次付费。

任务：

- 首页；
- 通话准备页；
- 智多星沟通页；
- 信息确认页；
- 销售信生成；
- 手机号授权拦截；
- 首次免费领取；
- 二次预览蒙版；
- 单封解锁；
- 年卡购买；
- 我的销售信；
- 我的档案基础版。

验收：

- 新用户首次生成后绑定手机号可看全文；
- 同手机号不能重复领取平台首次免费；
- 二次生成只能看预览；
- 单封支付后当前信件可看全文；
- 年卡支付后新生成信件可直接看全文。

### 阶段 2：总后台基础版

目标：总后台能控制主小程序和商业闭环。

任务：

- 管理员登录；
- 系统开关；
- 用户列表；
- 信件列表；
- 订单列表；
- 支付回调日志；
- 权益流水；
- 模板配置；
- 智多星话术配置；
- AI 供应商配置；
- 价格配置；
- 审计日志。

验收：

- 后台能关闭新生成能力；
- 后台能关闭支付入口；
- 后台能查看某用户的权益来源；
- 后台能看到每笔支付回调处理结果；
- 后台修改价格/模板后，小程序读取最新配置。

### 阶段 3：质量、风控与上线准备

目标：让一期可以小范围真实投放。

任务：

- 生成失败重试；
- 支付回调幂等测试；
- 订单补偿查询；
- AI 调用失败降级；
- 内容合规基础拦截；
- 年卡异常使用限流；
- 错误日志；
- 隐私协议和用户协议；
- 小程序体验版测试；
- 真机兼容测试。

验收：

- 支付重复回调不重复发权益；
- AI 失败不会丢失用户已采集信息；
- 总后台能定位失败订单；
- 小程序提交审核材料齐备。

## 9. 当前需要马上确认的事项

1. 小程序主体是否已经具备微信支付商户号。
2. 小程序类目是否允许使用实时音视频能力。
3. 是否已有 CloudBase 环境，还是需要新建。
4. 总后台技术栈偏好：React 还是 Vue。
5. 智多星声音一期是否用供应商默认音色，还是要定制音色。
6. 销售信生成主模型一期优先用哪家：通义千问、豆包、腾讯混元、DeepSeek，还是先做多供应商可切换。

## 10. 下一步建议

先做阶段 0，不直接进入大规模页面开发。

阶段 0 交付物：

1. 小程序基础工程；
2. CloudBase 后端基础工程；
3. 总后台基础工程；
4. 数据库集合/表设计；
5. 微信支付真机测试；
6. 实时语音 POC 对比；
7. 一期最终技术选型确认表。

完成阶段 0 后，再进入用户闭环开发。这样可以避免后面才发现支付、手机号、实时音视频权限或模型延迟不适合。
