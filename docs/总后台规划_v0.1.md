不阻塞开发、后续可在对应页面内补齐的组件状态：

- 客服/反馈入口。

结论：用户端不缺新的关键页面。接下来可以规划总后台。

## 2. 总后台一期定位

总后台不是给终端用户看的工具，而是运营和管理主小程序的控制台。

一期总后台要做到：

- 控制主小程序是否可用；
- 配置首页、通话引导、价格、权益、协议等关键内容；
- 查看用户、会话、销售信、档案、订单、权益流水；
- 处理生成失败、支付异常、权益异常；
- 查看调用成本、错误日志和支付回调；
- 为后续 OEM 保留 `tenant_id`，但不做多代理商管理页面。

一期总后台不做：

- 代理商入驻；
- 多小程序一键发布；
- 代理商结算；
- 复杂 BI 大屏；
- 人工给用户乱改权益；
- 人工绕过订单发年卡或单封权益。

权益必须由订单流水和权益流水驱动，后台只能查看、补偿和审计，不能让运营随手改前端权限。

目标：让运营快速知道今天主小程序是否正常。

### 5.3 通话引导配置

管理通话页内的问题、选项和顺序。

当前至少需要三个阶段：

1. `recipient_scope`：这封信帮谁写；
2. `letter_goal`：这封信想达成什么效果；
3. `profile_match`：是否匹配已有产品档案。

每个阶段字段：

- 阶段 key；
- 是否启用；
- 排序；
- 问题文案；
- 展示方式：通话卡片 / 候选弹层 / 自动判断；
- 是否必答；
- 是否允许语音回答；
- 是否允许跳过；
- 快捷选项；
- 选项业务值；
- 不确定时追问文案；
- 写入位置：本次项目 / 临时产品档案 / 长期产品档案 / 客户档案；
- 适用用户：新用户 / 单产品用户 / 多产品用户 / 免费用户 / 年卡用户。

### 5.4 销售信模板管理

管理“收集到的信息最终按什么结构写成销售信”。

通话引导只负责问清楚，信息确认只负责核对，模板管理负责决定写作结构、语气、段落要求和禁用表达。没有模板管理，生成质量不可控。

模板至少包含：

- 模板名称；
- 适用写信目标：回复 / 预约 / 购买 / 合作 / 复购 / 建立信任；
- 适用场景：我的产品 / 公司产品 / 朋友客户产品 / 临时项目；
- 输出长度：短 / 中 / 长；
- 语气风格：真诚、亲切、专业、克制、有推动力；
- 信件结构：开头、共鸣、问题、解决方案、证据、行动引导、结尾；
- 每段写作要求；
- 禁用表达；
- 必需素材：案例、价格、客户顾虑、证明、行动入口；
- 匹配规则；
- 是否启用；
- 版本号；
- 测试预览。

模板匹配逻辑：

```text
写信目标 + 产品归属 + 目标客户 + 用户补充信息 -> 匹配启用模板 -> 生成销售信
```

示例模板：

- 微信私聊成交信；
- 老客户复购信；
- 预约沟通信；
- 合作代理邀请信；
- 信任建立信；
- 价格解释信。

注意：

- 模板不等于前台用户可见的“模板库”；
- 用户端不展示“模板”概念；
- 模板是后台控制生成质量的结构化规则；
- 模板修改必须记录版本和审计日志；
- 已生成的旧信件要保留当时使用的模板版本。

- 财务；
- 客服；
- 只读查看。

权限原则：

- 财务能看订单和支付，不一定能改通话配置；
- 运营能改文案和配置，不一定能做支付补偿；
- 客服能看用户和信件状态，不一定能看完整支付回调；
- 所有敏感操作进审计日志。

## 6. 一期后台开发优先级

### P0：必须先做

- 管理员登录；
- 系统开关；
- 小程序配置；
- 通话引导配置；
- 销售信模板管理；
- 用户管理；
- 销售信管理；
- 订单与支付；
- 权益流水；
- 支付回调日志；
- 审计日志。

### P1：用户闭环稳定后补

- 档案管理；
- 会话与转写；
- 销售信生成配置；
- 价格与套餐；
- 协议与内容；
- 供应商配置。

### P2：一期后置

- 复杂运营数据；
- 成本趋势；
- 自动异常告警；
- 多租户页面；
- 代理商后台；
- OEM 发布。

## 7. 后台和用户端的关键关系

用户端不能写死的内容：

- 首页主文案；
- 首页按钮文案；
- 首页首次免费提示；
- 首页未领取提醒；
- 首页痛点区和三步流程文案；
- 首页插画/主视觉资源；
- 通话引导问题；
- 快捷选项；
- 引导阶段顺序；
- 销售信模板结构；
- 模板匹配规则；
- 写作段落要求；
- 禁用表达；
- 价格；
- 权益说明；
- 支付模式；
- 协议链接；
- 手机号授权文案；
- 生成失败文案。

用户端可以本地固定的内容：

- 页面基本布局；
- 通话页三个底部按钮；
- 微信系统授权触发逻辑；
- 基础状态样式；
- 已确认的“微信亲和绿”视觉方向。

## 8. 开发前需要确认

1. 总后台技术栈：React + Ant Design，还是 Vue + Element Plus？
2. 测试号阶段是否先做 mock 登录后台，还是直接做管理员账号密码登录？
3. 价格是否先按单封 200、年卡 2000 写入 mock 配置？
4. 供应商配置一期是否只保留字段，不先接多供应商切换？
5. 销售信模板第一批先内置哪些：成交信、复购信、预约沟通信、合作代理信是否够？
6. 后台第一版是先画 UI 审核稿，还是直接做可运行后台骨架？

## 9. 备注

新增任何依赖 CloudBase 数据库的后台功能前，需要先列出必须手动创建的集合、索引、权限和初始配置。

可能超过 60 秒的生成、补偿、导出、统计任务，必须任务化，不做前端同步等待。
