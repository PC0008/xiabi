# 《下笔有元》数据模型 v0.1

> 目标：把用户端页面、总后台 P0 页面、生成链路、支付权益链路落到 CloudBase 可实现的数据结构。  
> 当前阶段：微信小程序测试号 + CloudBase + mock/正式模式可切换。  
> 核心原则：后台可控配置不写死，长任务任务化，权益只认订单流水和权益流水。

## 1. 设计原则

1. 所有业务集合保留 `tenant_id`，一期只有主小程序也要保留。
2. 小程序端不直接决定权限、价格、模板、引导问题和权益。
3. 首页文案、通话引导、销售信模板、价格套餐、协议内容必须从后台配置读取。
4. 销售信生成、语音处理、导出等超过 60 秒风险的流程必须使用任务记录 + 轮询 + 重试。
5. CloudBase 数据库集合需要手动创建；开发前必须先创建集合、索引、权限和初始化配置。
6. 支付回调必须幂等；重复回调不能重复发权益。
7. 生成失败不能丢失用户已采集信息。
8. 后台所有敏感操作写入审计日志。

## 3. 通用字段

除明确说明外，业务集合建议包含：

```json
{
  "_id": "CloudBase 自动 ID 或业务 ID",
  "tenant_id": "main",
  "status": "active",
  "created_at": 1710000000000,
  "updated_at": 1710000000000,
  "created_by": "system",
  "updated_by": "system"
}
```

时间统一使用毫秒时间戳。展示时再由前端格式化。

## 9. 权限规则建议

CloudBase 数据库权限建议：

- 小程序端不直接写任何核心集合。
- 小程序端读取配置也优先通过云函数读取，便于过滤敏感字段和做缓存。
- 管理后台所有读写都通过云函数校验管理员身份。
- 支付回调只允许支付回调云函数写 `orders`、`payment_notify_logs`、`entitlement_ledgers`。
- 供应商密钥只存在服务端环境变量或密钥配置，不写入前端可读字段。

建议所有集合权限先设为：

```text
仅服务端可读写
```

如确需小程序端直读非敏感配置，也必须只读且只限发布态配置；一期建议先全部走云函数。

## 13. mock 与正式模式

需要支持 mock/正式切换：

- 支付：`mock` / `wechat`
- 语音转写：`mock` / 实际供应商
- 语音播报：`mock` / 实际供应商
- 销售信生成：`mock` / 实际供应商

模式配置放在：

- `pricing_configs.payment_mode`
- `provider_configs`
- `app_configs` 或 `system_switches` 中的入口开关

mock 阶段也必须写入真实业务集合，避免后面正式接入时重构数据流。

## 14. 关键不变量

1. 首页正式文案不写死，来自 `app_configs`。
2. 通话引导不写死，来自 `guide_configs`。
3. 销售信写作结构不写死，来自 `sales_templates`。
4. 价格不写死，来自 `pricing_configs`。
5. 用户是否可看全文由 `entitlement_ledgers` 计算。
6. 支付回调重复到达不能重复发权益。
7. 生成任务失败后，`conversation_sessions`、`transcript_segments`、`sales_letters` 不能丢。
8. 帮朋友/客户写信时，默认进入临时项目或朋友/客户产品档案，不污染用户自己的产品档案。
9. 后台每次修改配置、模板、价格、开关、权益补偿，都必须写入 `audit_logs`。
