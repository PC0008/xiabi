# 《下笔有元》开工前置清单

## 1. 先不要急着写业务代码

当前最重要的是把以下四件事定下来：

1. 项目框架和协作规则。
2. 用户端 UI 主流程。
3. 数据模型和权限边界。
4. API 契约和 mock 策略。

这些定好后再进入正式开发，能避免后期反复推倒页面、接口和数据结构。

## 2. UI 需要先确认

建议先做一份可打开的 HTML 原型，覆盖一期关键页面：

- 首次首页；
- 通话准备/按住说话页；
- 信息收集确认页；
- 销售信排队生成页；
- 手机号授权页；
- 销售信详情页；
- 二次使用预览蒙版页；
- 单封解锁/年卡购买页；
- 我的销售信；
- 我的档案基础版；
- 个人中心。

确认 UI 后，再把小程序页面按原型实现。

## 3. 架构需要先确认

一期推荐结构：

- 小程序：原生微信小程序。
- 后端：CloudBase 云函数为主。
- 长耗时/后续语音网关：CloudBase 云托管。
- 总后台：Web 工程，后续放在 `admin-web/`。
- 数据：CloudBase 数据库集合。
- 文件：CloudBase 云存储。

CloudBase 约束：

- 云函数单次执行时间按 60 秒上限设计，长任务不要同步等待。
- 销售信生成、语音识别/合成、PDF 导出、订单补偿等流程要预留任务表和状态轮询。
- 数据库集合需要在控制台手动创建；每新增集合、索引、权限或初始配置，都要先列清单给用户创建。

## 4. Mock 策略

测试号阶段先做 mock，不阻塞业务流：

- `phoneMode`: `mock` / `wechat`
- `paymentMode`: `mock` / `wechat`
- `aiMode`: `mock` / `provider`
- `voiceMode`: `mock` / `provider`

所有模式通过配置切换，不能写死在页面里。

## 5. 第一轮正式开发顺序

1. 定 UI 原型。
2. 定数据模型。
3. 定 API 契约。
4. 实现小程序主流程页面。
5. 实现 CloudBase mock API。
6. 接入真实 ASR/TTS/模型。
7. 切正式小程序后验证手机号和微信支付。

每进入一个需要数据库的开发步骤前，先输出对应的 CloudBase 集合创建清单。
